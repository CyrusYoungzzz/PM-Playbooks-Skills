# 贡献指南

本仓库是一组「产品方法论 → /command 路由」的集合。维护者参考此文档扩展。

## 怎么用（使用示例）

在 Claude Code 里输入命令，后接你要分析的对象：

```
/真伪需求分析 我想给陪聊 App 加一个"AI 自动给好友送语音祝福"功能，用户可以一键定时给联系人发周年纪念祝福
```

```
/爆品立项 做一款主打"零延迟"的开源蓝牙耳机，定价 199，瞄准学生群体
```

```
/预期诊断 我们的 PWA 工具上线 3 个月，DAU 稳定但分享率从 8% 跌到 2%，用户访谈反馈"挺好用的"
```

命令会读取对应的 reference 文档，按方法论的固定流程逐项分析，最后给出**可执行的下一步**而不是空话。

## 设计原则

按 `references/` 里的方法论沉淀 + `.claude/commands/` 里的命令路由分离结构（参考 [datawhalechina/hello-agents 的 Skill 最佳实践](https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra08-%E5%A6%82%E4%BD%95%E5%86%99%E5%87%BA%E5%A5%BD%E7%9A%84Skill.md)）：

1. **按需加载**：命令文件只放"流程指挥"，方法论本身放 `references/`，命令第一步加载它。避免一次性把所有方法论塞进上下文。
2. **命令是路由，不是知识**：命令文件本身要短，重点是规定"分析步骤" + "输出结构"，让 Claude 严格走流程不放飞自我。
3. **每个命令对应一个使用场景**，不要做"全能命令"。场景具体了，方法论的命中率才高。
4. **必须给可执行的下一步**：禁止"加油你行的"式空话，要落到"本周做哪个调研 / 砍掉哪个功能 / 改哪个文案"。
5. **如果方法论之间有冲突**，由命令决定调用哪个；reference 文档之间互不引用，避免循环依赖。

## 怎么扩

新增一个方法论时：

1. 把原始素材放到 `<人名>-raw/`（爬下来的文章 / 视频字幕 / 书摘）
2. 蒸馏出一个 `references/<人名>-<主题>.md`（按方法论的天然结构组织目录，开头标注来源）
3. 为每个使用场景写一个 `.claude/commands/<场景名>.md`，每个命令必须：
   - 第一步加载对应 reference
   - 把 `$ARGUMENTS` 代入框架
   - 规定输出结构
   - 列出反模式自检
4. 在 [SKILL.md](./SKILL.md) 的「当前收录」「命令清单」表格里追加一行
